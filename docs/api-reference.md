# ICE API Reference

Built for the Nomba Hackathon 2026 - Infrastructure Track.

This is a comprehensive reference for every endpoint currently implemented in ICE, written for anyone testing the API directly - via Swagger, Postman, curl, or a custom frontend. Alongside the request/response shapes, it documents non-obvious behavior discovered while building and testing the platform: which fields Nomba's real webhooks actually use, why an invoice never stays in draft, and a few auth-tier quirks that aren't obvious from the route names alone.

This file is also served live at GET /api-reference on the deployed API (see section 10) for use by the dashboard/frontend or anyone browsing directly.

For the product vision, see ICE_PRD.md. For engineering standards and git workflow, see ICE_ENGINEERING.md. For the live webhook delivery investigation specifically, see docs/webhook-integration.md.

---

## 1. Concepts

ICE has a three-level tenant hierarchy:

    Merchant (platform owner)
      -> Vendor (sub-account, own Nomba Dedicated Virtual Account)
           -> Customer (end customer of a vendor, optional own DVA)

- Merchant - the top-level account. Register once to get a master_api_key. This key manages vendors, views webhook deliveries, and accesses merchant-tier-only endpoints (misdirected payments, platform summary).
- Vendor - a sub-account under a merchant. Each vendor gets its own Nomba DVA at creation. Vendors have their own scoped vendor_api_key, generated by the merchant, used for day-to-day operations (invoices, customers).
- Customer - an end customer of a vendor. Can optionally get their own DVA. Invoices reconcile against the customer's own DVA, not the vendor's - see the Gotchas section, this is the single most common source of confusion when testing.

Core objects:
- Invoice - a billing request for a specific customer, with an amount in kobo. Auto-issued at creation (see Gotchas).
- Transaction - a record of an inbound payment (from a real or simulated Nomba webhook).
- Reconciliation - the automatic process matching a transaction to an invoice (exact match, overpayment, underpayment) or flagging it as unmatched.
- Misdirected Payment - a payment that arrived at a real, known virtual account but couldn't be matched to any open invoice. Reviewed manually: matched to an invoice retroactively, or refunded to the sender.
- Webhook Delivery - an outbound webhook ICE sends to a merchant's own webhookUrl (distinct from the inbound webhook ICE receives from Nomba).

## 3. Authentication

Two independent tiers of API key, both sent the same way:

    Authorization: Bearer <key>

| Tier | Variable name (Postman) | How to get it | Scope |
|---|---|---|---|
| Master (merchant) | master_api_key | Returned once from POST /v1/merchants/register, or after POST /v1/merchants/{id}/api-keys/rotate | Manage vendors, view webhook deliveries, misdirected payments, platform summary, mark-paid override |
| Vendor (scoped) | vendor_api_key | Returned once from POST /v1/vendors/{id}/api-keys, called with the master key | Create/view invoices, manage customers, view statements |

Both keys are shown exactly once - save them immediately from the response. There's no "view again" endpoint; if lost, rotate (master) or re-generate (vendor) to get a new one, invalidating the old.

A handful of endpoints unexpectedly require the master key despite being vendor-scoped resources (see Gotchas) - check the per-endpoint auth column below rather than assuming vendor-scoped always means vendor key.

### Response envelope

Every response follows the same shape:

    Success:
    { "ok": true, "data": { ... }, "requestId": "..." }

    Error:
    { "ok": false, "errorCode": "SOME_ERROR_CODE", "message": "Human-readable message", "requestId": "..." }

requestId is present on every response - useful for correlating with Render logs when something fails unexpectedly.

## 4. Suggested Testing Flow

The natural order to exercise the whole system, start to finish:

1. GET /healthz - confirm the service is up
2. POST /v1/merchants/register - get master_api_key
3. POST /v1/vendors - get vendor_id (provisions a real Nomba DVA)
4. POST /v1/vendors/{id}/api-keys - get vendor_api_key
5. POST /v1/vendors/{id}/customers - get customer_id (optionally set provisionDva: true here)
6. POST /v1/vendors/{id}/customers/{cid}/account - provision the customer's own DVA, if not done in step 5. This step is easy to skip and is required before any invoice for this customer can ever reconcile.
7. POST /v1/invoices - create and auto-issue an invoice against that customer
8. Either simulate a webhook (see section 8) targeting the customer's DVA number with a matching amount, or manually override with POST /v1/invoices/{id}/mark-paid
9. Check GET /v1/invoices/{id}/reconciliation, GET /v1/vendors/{id}/statement, GET /v1/vendors/{id}/transactions to see the result

## 5. Endpoint Reference

### Health

| Method | Path | Auth | Notes |
|---|---|---|---|
| GET | /healthz | none | Mounted at app root, not under /v1. Returns {"ok":true,"redis":"ready"}. |

### Merchants

| Method | Path | Auth | Notes |
|---|---|---|---|
| POST | /v1/merchants/register | none | Body: businessName, email, webhookUrl. Returns merchant plus api_key - api_key shown once. |
| GET | /v1/merchants/{id} | master | Merchant profile. |
| PUT | /v1/merchants/{id}/webhook-url | master | Body: webhookUrl. Updates the merchant's own outbound webhook target. |
| POST | /v1/merchants/{id}/api-keys/rotate | master | Old key invalidated immediately. New key returned once. |
| GET | /v1/merchants/{id}/webhook-deliveries | master | List of outbound webhook delivery attempts. |
| GET | /v1/merchants/{id}/summary | master | Platform-wide rollup - see shape below. |

Platform summary response shape:

    {
      "total_collected_kobo": "1500000",
      "total_vendors": "1",
      "active_vendors": "1",
      "misdirected_count": "6",
      "overpayment_count": "0",
      "underpayment_count": "1",
      "exact_match_count": "1",
      "total_transactions": "8",
      "refunds_issued_kobo": "0",
      "reconciliation_rate_percent": 12.5
    }

Note: refunds_issued_kobo currently only counts the overpayment auto-refund queue path. A misdirected payment refunded via POST /payments/{id}/refund (which calls Nomba's Transfer API directly) does not currently increment this figure.

### Vendors

| Method | Path | Auth | Notes |
|---|---|---|---|
| POST | /v1/vendors | master | Body: name. Provisions a real Nomba DVA server-side. Vendor keys get a 403 here. |
| GET | /v1/vendors | master | Query: page, pageSize, status (pending/active/suspended). |
| GET | /v1/vendors/{id} | master | Vendor profile, including nomba_va_number, nomba_bank_name, callback_url. |
| POST | /v1/vendors/{id}/api-keys | master | Generates a scoped vendor key, shown once. |
| POST | /v1/vendors/{id}/account/suspend | master | Suspends the vendor account. |
| PUT | /v1/vendors/{id}/account | master (not vendor - see Gotchas) | Body: name and/or callbackUrl, both optional. callbackUrl must start with https://. |

### Customers (nested under vendors)

| Method | Path | Auth | Notes |
|---|---|---|---|
| POST | /v1/vendors/{id}/customers | vendor | Body: name, email, provisionDva. Set provisionDva: true to get a DVA immediately. |
| GET | /v1/vendors/{id}/customers/{cid} | vendor | Customer profile. |
| POST | /v1/vendors/{id}/customers/{cid}/account | vendor | Provisions a DVA if the customer was created with provisionDva: false. This is the account invoices actually reconcile against - see Gotchas. |

### Invoices

| Method | Path | Auth | Notes |
|---|---|---|---|
| POST | /v1/invoices | vendor | Body: vendor_id, customer_id, amount_kobo. customer_id optional. Auto-issues on creation - returns status "issued", not "draft". |
| GET | /v1/invoices/{id} | vendor | Invoice details. |
| GET | /v1/invoices/vendor/{vendorId} | vendor | List invoices for a vendor. |
| PATCH | /v1/invoices/{id}/issue | vendor | Transitions draft to issued. Since invoices auto-issue on creation, calling this on the same invoice again will 400 with INVALID_TRANSITION (issued has no self-transition). |
| POST | /v1/invoices/{id}/mark-paid | master | Manual override. Platform-owner-only, audit-logged. |
| GET | /v1/invoices/{id}/reconciliation | vendor | Reconciliation status for this one invoice. |
| GET | /v1/invoices/reconciliation/logs | master | Platform-wide reconciliation log, all invoices. Query: page, limit. Distinct from the per-invoice endpoint above. |

### Misdirected Payments

| Method | Path | Auth | Notes |
|---|---|---|---|
| GET | /v1/payments/misdirected | master only | Query: page, limit. Payments that landed on a known VA with no matching invoice. |
| POST | /v1/payments/{id}/match | master | Body: invoiceId. Retroactively reconciles the payment against an invoice (exact/over/underpayment logic applies). |
| POST | /v1/payments/{id}/refund | master | Refunds the payment to the original sender via a real Nomba Transfer API call (account lookup, then transfer). |

### Webhook Deliveries (outbound)

| Method | Path | Auth | Notes |
|---|---|---|---|
| POST | /v1/webhook-deliveries/{id}/replay | master | Reads the delivery ID from the JSON request body ({"id": "..."}), not the URL path parameter, despite the route pattern - send both. |

### Statements and Reporting

| Method | Path | Auth | Notes |
|---|---|---|---|
| GET | /v1/vendors/{id}/statement | vendor | Query: page, pageSize, from, to, status. |
| GET | /v1/vendors/{id}/customers/{cid}/statement | vendor | Same query params, scoped to one customer. |
| GET | /v1/vendors/{id}/transactions | vendor | Query: page, pageSize. Raw transaction records for a vendor. |
| GET | /v1/merchants/{id}/summary | master | See Merchants section above. |
| GET | /v1/transactions/{id} | vendor | Single transaction by its internal id (not Nomba's transactionId string - see Gotchas). |

### Webhooks (Nomba Inbound)

| Method | Path | Auth | Notes |
|---|---|---|---|
| POST | /v1/webhooks/nomba | HMAC signature, not Bearer | Headers: nomba-signature, nomba-timestamp. See section 6 below for the full payload shape and signing scheme. |

### Non-POST to the webhook endpoint

GET (or any non-POST) to /v1/webhooks/nomba returns 405, confirming the routing fix that previously caused non-POST methods to hit an auth-gated router and return a misleading 401 instead.

## 6. Real Nomba Webhook Payload Shape

This is the actual, confirmed shape Nomba sends for a payment_success event, verified against Nomba's own docs and this project's HMAC implementation:

    {
      "event_type": "payment_success",
      "requestId": "<uuid>",
      "data": {
        "merchant": {
          "userId": "...",
          "walletId": "...",
          "walletBalance": 732233.66
        },
        "terminal": {
          "terminalId": "",
          "terminalLabel": ""
        },
        "transaction": {
          "transactionId": "...",
          "type": "vact_transfer",
          "time": "2026-07-12T12:00:00Z",
          "responseCode": "00",
          "transactionAmount": 15000,
          "fee": 5,
          "aliasAccountNumber": "5035753011"
        },
        "customer": {
          "senderName": "JOHN DOE",
          "accountNumber": "0123456789",
          "bankName": "Some Bank",
          "bankCode": "044"
        }
      }
    }

Key fields the app actually reads:

| Field | Path | Purpose |
|---|---|---|
| Amount | data.transaction.transactionAmount | In Naira, not kobo (confirmed: walletBalance in the same payload has decimal places, which kobo, an indivisible subunit, cannot have). Converted to kobo internally. |
| Receiving account | data.transaction.aliasAccountNumber | The virtual account the payment landed in - this is what reconciliation matches against, either a customer's or a vendor's DVA number. |
| Sender name/account/bank | data.customer.senderName, accountNumber, bankCode | Used for misdirected-payment refunds. |

### Signature verification

Headers: nomba-signature (base64 HMAC-SHA256), nomba-timestamp (Unix seconds, must match what was signed).

Signed string, colon-joined:

    event_type:requestId:merchant.userId:merchant.walletId:transaction.transactionId:transaction.type:transaction.time:transaction.responseCode:timestamp

responseCode defaults to an empty string if absent. The result is HMAC-SHA256'd with the shared secret and base64-encoded.

## 7. Known Implementation Quirks and Gotchas

A few behaviors that aren't obvious from the route names or schema alone, discovered while testing:

1. Invoices reconcile against the customer's own DVA, not the vendor's shared VA. The invoice-matching query joins only through the customers table. A payment sent to the vendor's own virtual account will never match any invoice, no matter what's open - it will always be flagged as a misdirected payment instead. If invoices never seem to reconcile during testing, check whether the customer actually has their own DVA provisioned (POST /vendors/{id}/customers/{cid}/account).

2. Invoices auto-issue on creation. POST /v1/invoices internally issues the invoice immediately, so the response always shows status "issued", never "draft". Calling PATCH /invoices/{id}/issue on that same invoice afterward will 400 with INVALID_TRANSITION.

3. PUT /vendors/{id}/account requires the master key, even though it's a vendor-scoped resource - the controller explicitly checks for and rejects vendor-tier keys here.

4. callbackUrl only reaches Nomba via the vendor update path, not at vendor creation. POST /vendors provisions the DVA but sends only accountRef, accountName, and currency - no callback URL. To get Nomba actually calling back for a given vendor's VA, a separate call to PUT /vendors/{id}/account with a callbackUrl is required after creation.

5. POST /webhook-deliveries/{id}/replay reads the ID from the JSON body, not the URL path parameter, despite the route pattern implying otherwise - send the id in the body as well as the path.

6. Misdirected payments only get flagged for known, registered virtual account numbers. A payment to a genuinely unregistered/made-up account number will fail to resolve a merchant internally and will not produce a misdirected-payment record at all - the app can only flag payments to VAs it already knows about (a vendor's or customer's real DVA) that simply have no invoice open against them right now.

7. GET /healthz lives at the app root, not under /v1 - the one exception to every other endpoint's base path.

## 8. Live Webhook Delivery Status

As of this writing, Nomba has not registered any webhook subscription for this hackathon sandbox account, confirmed directly via Nomba's own /v1/webhooks/events and /v1/webhooks/event-logs APIs (both return empty despite two onboarding form submissions). This means real inbound webhooks from Nomba have never been delivered to this deployment - every transaction, reconciliation, and misdirected-payment record described in this document has been produced via manually-signed simulated webhooks (see the Postman collection's "Webhooks (Nomba Inbound)" folder), not live Nomba traffic.

The entire signature verification, payload validation, reconciliation, and misdirected-payment pipeline has been independently verified end-to-end using this simulated approach. See docs/webhook-integration.md for the full investigation and the test harness details.
